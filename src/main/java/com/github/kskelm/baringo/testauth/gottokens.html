Step 3: Save your tokens!

And here we are... we have an access_token and a refresh_token. Here they are:

Access token:{access_token}
Refresh token:{refresh_token}
Expires in:{expires_in} seconds
Expires on:{expires_on}
Token type:{token_type}
Scope:{scope}
User id:{user_id}
User name:{user_name}

Baringo is not, in itself, responsible for fetching and storing your authorization_code, your access_token, or your refresh_token. You need to store those (hopefully encrypted) yourself someplace else, probably in a database.

In Baringo, one BaringoClient... 

        BaringoClient client = new BaringoClient( client_id, client_secret );

...can either access resources anonymously, or you can attach an account to it by calling one of the two approaches. In the first case, you can pass your authorization_code to Baringo and it will attempt to automatically maintain the tokens for you: 

            client.setAuthenticationCode( authentication_code );

The down side is that authorization codes live only a few minutes! You need to get the code into Baringo immediately so it can generate the access and refresh tokens, or you need to do that yourself. An advantage to doing this yourself is that you should be the one to store the tokens anyway, encrypted in a database somewhere. More about that later.
